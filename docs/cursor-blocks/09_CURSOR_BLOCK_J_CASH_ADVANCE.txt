═══ ITEM 2 — BACKEND SERVICES ═══

Create apps/backend/src/driver-finance/cash-advances.service.ts:

  Driver-facing operations:
    createRequest({driver_id, lines, payment_method, urgency, notes, 
                   related_load_id?, receipts?})
      - Validates personal-per-load rule
      - Validates weekly personal cap (config-driven)
      - Validates sum-of-lines matches total
      - Auto-suggests related_load_id from driver's active dispatches
      - Generates request_number (CA-YYYY-NNNNNN)
      - Creates audit event 'requested'
      - Sends notifications: PWA push to office (accountants + dispatchers)
      - Sends WhatsApp to driver (confirmation)
    
    getMyRequests(driver_id, filter)
    cancelMyRequest(request_id, driver_id) — only if status=pending_review
    
  Office-facing operations:
    listPendingReview(filters) — for accounting/dispatcher inbox
    getRequest(id) — includes lines + recoveries + audit log
    
    reviewRequest(request_id, {
      line_adjustments: [{line_id, approved_amount_cents}],
      approval_notes,
      paid_from_gl_account_id,
      withdrawal_code,
      withdrawal_code_provider,
      action: 'approve' | 'partial_approve' | 'deny',
      denied_reason?
    })
    
      - Validates approver role (Owner, Administrator, Accountant, Dispatcher)
      - For approval:
        - Calculates approved_total_cents
        - Determines accounting_treatment:
          * If related_load_id is set AND load.status IN ('assigned', 
            'in_transit', 'delivered') → 'bill_payment'
          * Else → 'driver_advance_asset'
        - Creates accounting records (see Item 3)
        - Creates audit event 'approved' or 'partial_approved'
        - Notifies driver: PWA + WhatsApp + Email
        - Queues QBO sync via outbox event
      - For denial:
        - Records denied_reason
        - Audit event 'denied'
        - Notifies driver: PWA + WhatsApp + Email
    
    adjustApprovedAmount(request_id, adjustments) — pre-payout only
    markPaidOut(request_id) — when payment actually executed
    
  Recovery operations:
    listOpenAdvancesForDriver(driver_id)
      - Returns all driver_advance_asset rows with status='approved' or 
        'recovered_partial' AND fully_recovered_at IS NULL
    
    applyRecovery({request_id, source: 'driver_bill' | 'settlement' | 
                   'manual', source_id, recovery_amount_cents, 
                   recovered_by_user_id, notes})
      - Validates recovery_amount <= (approved_total - recovered_total)
      - Inserts recoveries row
      - Updates request's recovered_total_cents
      - If recovered_total == approved_total: sets fully_recovered_at
      - Creates GL entries (Driver Pay Expense ← Driver Advances Asset)
      - Audit event 'recovered_partial' or 'recovered_full'
      - Queues QBO sync

═══ ITEM 3 — ACCOUNTING INTEGRATION ═══

Hook into existing accounting flows:

A) On approve → create accounting records:
   - If 'bill_payment': 
     - Insert row in accounting.bill_payments referencing driver_bill for 
       this load (find via mdata.loads → driver_finance.driver_bills)
     - Sets payment_type='cash_advance_to_driver'
     - Sets amount = approved_total_cents
   
   - If 'driver_advance_asset':
     - Create GL journal entry:
       Debit: catalogs.accounting_gl_accounts where code='DRIVER_ADVANCES_RCV'
       Credit: paid_from_gl_account_id (Comdata Cash, etc.)
     - Records driver_advance_asset_gl_entry_id

B) Bill creation hook (existing accounting.bill creation flow):
   - Add hook in apps/backend/src/accounting/bills.service.ts
   - After bill created for a driver:
     - Call listOpenAdvancesForDriver(driver_id)
     - If any open advances exist: return them in bill creation response 
       under field 'driver_open_advances' for frontend to show prompt
   - Add new endpoint POST /api/v1/accounting/bills/:id/apply-advances 
     that accepts list of {advance_request_id, amount_cents} and calls 
     applyRecovery for each

C) Settlement generator hook:
   - In existing settlement preview generator:
     - For each driver: call listOpenAdvancesForDriver
     - Include all open advances as line items on settlement preview
     - When settlement is approved: auto-call applyRecovery for each 
       advance with source='settlement'

D) QBO sync hook:
   - Emit outbox event 'driver_finance.cash_advance.approved' on approval
   - Emit outbox event 'driver_finance.cash_advance.recovered' on recovery
   - Create QBO sync handlers:
     - cash_advance.approved → create QBO Bill Payment OR Journal Entry
     - cash_advance.recovered → create reclassification Journal Entry
