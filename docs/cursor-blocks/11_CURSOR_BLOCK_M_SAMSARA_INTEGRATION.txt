Backend services:
  apps/backend/src/integrations/samsara/connection.service.ts
    connect({operating_company_id, api_token, user_id})
      - Encrypts token with AES-256 (SAMSARA_TOKEN_ENCRYPTION_KEY env var)
      - Tests via GET /me
      - Stores connection
      - Returns org_id + account_name
    
    disconnect(connection_id, user_id)
    getConnection(operating_company_id)
    testConnection(operating_company_id) — calls /me, returns health
  
  apps/backend/src/integrations/samsara/api-client.service.ts
    Reusable client:
      - Loads decrypted token
      - Adds Authorization header
      - Retry with exponential backoff
      - Honors X-RateLimit-Remaining header
      - Audits every call to samsara_api_audit
  
  apps/backend/src/integrations/samsara/driver-sync.service.ts
    initialBulkImport(operating_company_id)
      - Pulls all drivers via GET /fleet/drivers (paginated)
      - For each:
        a) Try exact match in mdata.drivers (by name + license_number)
        b) If no exact: fuzzy match (Levenshtein < 3 on name + license prefix)
        c) If still no match: create new mdata.drivers row with 
           status='pending_office_review'
        d) Mark active=true if Samsara isDeactivated=false
        e) Try to find QBO vendor:
           - Exact match on display_name
           - Fuzzy match on display_name
           - Store qbo_vendor_id if found
           - Queue for manual review if not
      - Insert samsara_driver_mappings row for each
      - Return summary: created, updated, manual_review_required
    
    pollDriverChanges(operating_company_id, since_timestamp)
      - Daily cron: pull drivers modified since last sync
      - Reconcile: active/deactivated changes
      - Update existing mappings, create new if needed
    
    listMatchReviewQueue(operating_company_id)
    resolveMatchReview(mapping_id, {ih35_driver_id, qbo_vendor_id, action})

═══ PR 2 OF BLOCK M — S3 + S4: VEHICLES + GEOFENCE/TRIP STATUS ═══

Branch: feat/p8d-samsara-vehicles-geofences

This is the OPERATIONAL CORE — geofence on every load with driver alerts. 
Critical for daily operations.

Migration 0126_p8d_samsara_vehicles_geofences.sql:
  - integrations.samsara_vehicle_mappings
  - integrations.samsara_vehicle_odometer_history
  - integrations.samsara_vehicle_location_history
  - integrations.samsara_load_geofences
  - integrations.samsara_geofence_events

Backend services:
  vehicle-sync.service.ts (similar pattern to driver-sync)
  
  geofence-engine.service.ts
    createGeofencesForLoad(load_id)
      Called from existing book-load flow
      For each load_stop:
        Call Samsara POST /fleet/geofences with:
          - circular shape, default 300m radius
          - name: "Load L-NNNN [pickup/delivery] [city]"
          - tags for traceability
        Store in samsara_load_geofences
    
    deleteGeofencesForLoad(load_id)
      Called when load.status changes to 'completed' or 'cancelled'
      Calls Samsara DELETE /fleet/geofences/{id} for each
      Marks deleted_at in DB
    
    handleGeofenceEntry(webhook_payload)
      Webhook handler — verifies HMAC signature first
      Looks up load_geofence by samsara_geofence_id
      Identifies driver via current vehicle assignment
      Inserts samsara_geofence_events row
      Triggers driver PWA notification:
        - Push notification with load context
        - Status prompt opens when driver taps
      Triggers office notification on dispatch board
    
    handleDriverStatusResponse(geofence_event_id, status, driver_user_id)
      Records driver_response + driver_responded_at
      Creates dispatch.load_events row
      Updates load.status if applicable

Hook into existing book-load flow:
  In apps/backend/src/dispatch/book-load.service.ts:
    After load + load_stops created successfully:
      AWAIT geofenceEngine.createGeofencesForLoad(load.id)
      If fails: log error but don't block load creation
      (loads work without geofences; geofences are an enhancement)

Hook into load-completion flow:
  When load.status changes to 'completed' or 'cancelled':
    AWAIT geofenceEngine.deleteGeofencesForLoad(load.id)

Webhook endpoint:
  POST /api/v1/integrations/samsara/webhook
  Verifies signature (HMAC SHA-256 with shared secret)
  Routes by event type to appropriate handler

Frontend (Driver PWA):
  apps/frontend/src/pages/driver/trip-status-prompt/:
    DriverTripStatusModal.tsx
      Triggered by geofence entry push notification
      Shows:
        - Load context (load number, customer, location)
        - Status buttons (depending on stop_type):
          * Pickup stop: Arrived / Loading / Loaded / Departed
          * Delivery stop: Arrived / Unloading / Checked-in / Delivered / Departed
      On select: POST /api/v1/driver/loads/:load_id/status
      Bilingual EN/ES based on driver.preferred_language

Frontend (Office):
  Update LoadDetailPage.tsx:
    Add "Geofence Timeline" tab showing:
      - All geofence entry/exit events for this load
      - Driver responses + timestamps
      - Map view (when S4 GPS phase merged)
