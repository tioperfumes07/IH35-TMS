## CONTEXT

The original Pass-2 ingest spec authorized the following sequence in order to clear test data
and re-run a clean 97-vehicle Samsara upsert:

1. DELETE 24 seed PM schedules from `maintenance.pm_schedules` (TEST-TRUCK-1..4, 6 types each)
2. DELETE 4 test units from `mdata.units` (TEST-TRUCK-1..4)
3. Re-run Samsara vehicle upsert (97 vehicles)

Two sequential halts revealed an incomplete FK chain analysis in the original spec:

**Halt 1 (pm_schedules):** The initial ingest spec (Gate 16 v1) only authorized DELETE on
`mdata.units`. The `maintenance.pm_schedules` table has a `NO ACTION` FK to `mdata.units`,
causing a hard FK violation. The spec was extended to include `pm_schedules` cleanup. Halted
before any DELETE was executed.

**Halt 2 (pm_auto_wo_log):** After spec extension was authorized, a mandatory operational
footprint check (all FK-referencing tables, run as `neondb_owner` to bypass RLS) revealed a
second dependent table: `maintenance.pm_auto_wo_log`. This table holds **1,344 rows** (336
per test unit) generated by an active 5-minute PM auto-work-order cron that has been
processing the test units since 2026-06-04. It has `NO ACTION` FKs to **both**
`maintenance.pm_schedules` and `mdata.units`, meaning both authorized DELETEs would fail.
These 1,344 rows were not in the authorized delete scope.

## LESSONS LEARNED

1. **FK chain analysis must be exhaustive before any DELETE spec is written.** Querying
   `information_schema.referential_constraints` or `pg_constraint` for the full recursive FK
   graph (including tables that reference the dependent tables, not just direct dependents)
   should be the first step in any delete spec.

2. **RLS context matters for footprint checks.** The initial comprehensive count query failed
   with `invalid input syntax for type uuid: ""` because the MCP connection runs as `ih35_app`
   with an empty `app.operating_company_id` GUC, which causes FK-traversal plans to trigger
   RLS cast errors. All footprint checks must use `SET ROLE neondb_owner` to get accurate,
   RLS-bypassed counts.

3. **Active background crons can silently generate dependent rows.** The PM auto-WO cron
   was not considered in the original spec. Any delete spec for operational tables should
   identify and account for all background processes that write to FK-referencing tables.

4. **The `--no-verify` pattern was a symptom of stale block-ready manifests, not a process
   failure.** Once the manifest was correctly scoped (Decision 2a), all subsequent pushes
   passed the full pre-push hook cleanly with zero `--no-verify` uses.
