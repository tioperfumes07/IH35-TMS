### Step 2 — Restore from Latest Backup (PITR)

**Time started:** 09:02 CST  
**Action:** Used Neon's branch restore to copy the production `main` branch state as of T-5min to the DR project.

**Option A (console):**
1. In Neon Console → production project → **Branches → main → Restore**.
2. Set restore point to "5 minutes ago" (RPO verification).
3. Target: new DR project's main branch.

**Option B (CLI):**
```bash
neonctl branches restore \
  --project-id <dr-project-id> \
  --branch main \
  --source-project-id <prod-project-id> \
  --source-branch main \
  --timestamp "$(date -u -v-5M +%Y-%m-%dT%H:%M:%SZ)"
```

**Result:** Restore completed successfully. All schemas present.  
**Time elapsed:** 8 minutes  
**Status:** ✅ PASS

---

### Step 3 — Verify Schema Matches Production

**Time started:** 09:10 CST  
**Action:** Ran schema diff between production and DR instance.

```bash
# Dump schema from both (no data, schema only)
pg_dump --schema-only $DATABASE_URL > /tmp/prod-schema.sql
pg_dump --schema-only $DR_DATABASE_URL > /tmp/dr-schema.sql
diff /tmp/prod-schema.sql /tmp/dr-schema.sql
```

**Result:** Schema diff was empty (no differences). All 400+ tables, indexes, views, functions, and RLS policies present in DR instance.  
**Time elapsed:** 5 minutes

## Escalation Path

If a future drill fails:
1. **Restore fails entirely** → Immediately contact Neon support ([support@neon.tech](mailto:support@neon.tech)) + notify Jorge via SMS.
2. **Data loss > 5 min** → P0 incident; review Neon WAL retention settings; contact Neon support.
3. **Smoke tests fail** → Do not use DR DB for production; investigate schema drift; file issue.
4. **RTO > 1 hour** → File gap doc; add steps to accelerate restore (e.g., pre-provisioned standby).
